# 画像判定 人力テスト計画書

作成日: 2026-07-08 ／ 対象: りなれす v6（gpt-4o Vision 画像判定）

## 1. 目的

デバイス写真の AI 判定（`/devices/new` の「種類をえらぶ」）が、**実際の撮影条件**でどの程度正しく・使いやすく動くかを人手で検証する。

自動評価（フリー素材25枚）では Top-1/Top-3 とも100%だったが、白背景・鮮明な公式写真中心のため、実写（生活感のある背景・手持ち・暗所・ジャンク品）での実力は未検証。そこを人力テストで埋める。

## 2. 体制・期間

| 役割 | 人数 | 内容 |
|---|---|---|
| テスター A / B / C | 3名 | 分担カテゴリの撮影・判定・記録 |
| とりまとめ（A が兼任） | 1名 | 進捗管理・集計・報告書ドラフト |

期間は **7日間**:

| 日 | 内容 |
|---|---|
| Day 1 | キックオフ（本計画書の読み合わせ・アカウント準備・記録シートの使い方確認・被写体の調達確認） |
| Day 2〜5 | テスト実施（1人あたり約5〜6判定/日。すきま時間で可） |
| Day 6 | 集計・不足ケースの追試（集計シートで自動計算→未消化ケースを潰す） |
| Day 7 | 結果レビュー会・報告（合否判定・改善要望リストの確定） |

## 3. テスト環境・アカウント

- **本番 Web** をスマートフォンのブラウザで使用: `https://ca-rinaresu-web.happywater-4b65aee9.japaneast.azurecontainerapps.io`
- 各自の Google アカウントでログインし、**あだ名の先頭に「テスト」**を付けて登録（例: テストたろう）→ テスト終了後の運営側データ整理を容易にするため
- **判定テストは「登録せず戻る」が基本**: 判定結果（候補と一致度%）を記録シートに書いたら、「この種類で進む」を押さずに「← もどる」で退出する。本番 DB にテストデバイスを溜めないため
  - 例外: フロー全体の確認ケース（各自1件だけ登録→送付→検収まで通してよい）
- **写真の注意**: 人物・住所・書類など個人情報が写り込まないこと。撮った写真はアップロードされサーバーに保存される前提で撮影する

## 4. 検証観点

1. **判定精度**: 正解が候補1位（Top-1）／候補3件以内（Top-3）に入る率
2. **紛らわしい品目の識別**: ガラケー vs スマホ、タブレット vs 大型スマホ、コンデジ vs 一眼
3. **対象外・非端末の安全性**: 財布・食品などを高い一致度で端末と誤判定しないか（誤ポイント付与リスク）
4. **撮影条件への頑健性**: 角度・明るさ・背景・ケース付き等での精度変化
5. **安定性**: 同一写真を3回判定して結果がブレないか
6. **応答時間・UX**: 判定待ち時間の体感、候補選択・手動選択の使いやすさ

## 5. 分担とテストケース

### 撮影条件コード（記録シートの「条件」列）

| コード | 条件 |
|---|---|
| ① | 正面・明るい場所・無地に近い背景（基本形） |
| ② | 斜め45度から |
| ③ | 近接撮影（画面や一部分のみ） |
| ④ | 暗い場所 |
| ⑤ | 背景が雑多（散らかった机の上など） |
| ⑥ | 手に持って撮影 |
| ⑦ | ケース・カバー付き、または付属品（ケーブル等）と一緒 |
| ⑧ | 安定性チェック: 同じ写真で3回連続判定（3行に分けて記録） |

### 分担（1人 = 担当2カテゴリ × 条件①〜⑦ + 安定性⑧×3回 + 対象外4件 = 21判定）

| テスター | 担当カテゴリ（各7条件） | 安定性⑧ | 対象外（非端末）4件 |
|---|---|---|---|
| A | スマートフォン／ガラケー | スマートフォンで実施 | 財布・食品・本・鍵など |
| B | タブレット／デジタルカメラ | タブレットで実施 | リモコン・電卓など**紛らわしい家電系** |
| C | 携帯ゲーム機／その他小型家電 | 携帯ゲーム機で実施 | 文房具・おもちゃなど |

- 合計 63判定（+ 余力があれば自由枠: 面白い誤判定を探す探索テスト。記録シートに空行あり）
- 被写体は自宅の実物を基本とし、無い品目は家電量販店の店頭・知人の物などで代替（撮影マナーに注意）。どうしても実物がなければ「PC 画面に表示した写真を撮る」は**最終手段**（その場合は備考に明記）

## 6. 記録方法

**記録シート**: [画像判定_テスト記録シート.xlsx](画像判定_テスト記録シート.xlsx)（同フォルダ）

- 「記録シート」タブに63ケースが事前入力済み。テスターは**実施日・候補1〜3位・一致度%・手動選択の有無・体感応答時間・備考**を埋めるだけ
- 「正解が1位？」「3位以内？」は自動判定（数式）
- 「集計」タブで Top-1/Top-3・カテゴリ別成績・対象外の誤検知・合否が**自動計算**される
- 3名での共有方法は任意（1ファイルを回す／各自コピーして Day 6 にとりまとめが結合）

## 7. 合格基準

| 項目 | 基準 |
|---|---|
| Top-3 正解率（主指標） | **90% 以上** |
| Top-1 正解率 | 75% 以上 |
| 対象外の誤検知（一致度70%以上で端末クラスと判定） | 5% 以下 |
| 安定性（⑧の3回で候補1位が全一致） | 3名とも「ブレなし」 |
| 体感応答時間 | 「5秒以上」が全体の2割以下 |

## 8. 集計・報告（Day 6〜7）

1. 集計タブの自動計算結果を確認し、未消化ケースを追試
2. 誤判定ケースを一覧化: どの品目が・どの条件で・何に間違えたか（混同の傾向）
3. UX 上の気づき（備考欄）を分類し、改善要望リストにまとめる
4. 合否判定と改善提案を添えて報告（不合格項目があれば、プロンプト改善→自動ハーネス（`backend/eval/`）で回帰確認→必要なら人力再試験、のループへ）

## 9. リスクと注意

- **本番環境でのテスト**のため、判定のみで登録しない運用を徹底する（§3）。誤って登録した場合は備考に記録し、Day 7 にまとめて報告
- gpt-4o の応答は完全な決定論ではないため、同一写真でも結果が変わり得る（それ自体が⑧の検証対象）
- 判定失敗時はモック判定（画像と無関係な候補）に自動フォールバックする仕様。**明らかに画像と無関係な候補ばかり**が出た場合はその旨を備考に書く（フォールバック発生の疑い）
